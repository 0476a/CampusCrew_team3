# CampusCrew AI 구현 가이드

## 1. 문서 목적

이 문서는 AI 담당자가 요구사항 문서를 모두 읽은 뒤, 4개 AI 기능을 어떤 구조와 순서로 구현할지 정리한 작업 지침서다.  
무엇을 만들어야 하는지는 `AI_SPEC.md`, `PRD.md`, `API_SPEC.md`에 있고, 이 문서는 그 범위를 어떤 방식으로 설계하고 연결할지 설명한다.

## 3. AI 역할

AI 담당자는 단순히 모델만 호출하는 역할이 아니다.

- 4개 기능의 입력과 출력 구조 설계
- 프롬프트 구조 설계
- 실패 처리 기준 정리
- 프론트와 백엔드가 다룰 수 있는 응답 형식 고정
- RAG용 문서 수집, 청크화, 검색, 출처 반환 흐름 설계
- 배포 환경에서 AI 기능 검수

## 4. 구현 대상 기능

- 회의록에서 액션 아이템 추출
- 문서 톤 변환
- 진행 상황 요약
- 팀 문서 Q&A

## 5. 고정 기술과 사용 기준

| 항목 | 기술 | 사용 기준 |
| --- | --- | --- |
| 모델 호출 | 팀이 선택한 단일 LLM API | 기능마다 모델을 바꾸지 않고, 호출 방식과 응답 형식을 통일한다. |
| 출력 구조 | Structured Output 또는 JSON 응답 | 프론트와 백엔드가 바로 다룰 수 있는 구조로 고정한다. |
| 검색 | 벡터 검색 | 팀 문서 Q&A에만 적용한다. |
| 임베딩 | 단일 임베딩 모델 | 문서 청크 임베딩과 질의 임베딩에 같은 기준을 사용한다. |
| 출처 반환 | 문서 ID, 제목, 일부 본문 | 답변만 주지 말고 어떤 문서를 근거로 썼는지 같이 반환한다. |

## 6. 구현 원칙

- 4개 기능 모두 입력과 출력 형식을 먼저 고정한다.
- 자유 문장 응답보다 구조화된 응답을 우선한다.
- 프롬프트는 기능별로 역할, 입력, 출력, 금지사항이 분명해야 한다.
- 실패 응답도 성공 응답만큼 중요하게 다룬다.
- RAG는 팀 문서 Q&A에만 적용하고, 나머지 기능은 문서 검색 없이 동작하게 둔다.
- AI 기능은 기존 화면 안에서 사용되는 도구로 설계한다.

## 7. 구현 시작 전 준비

### 7.1 먼저 정해야 하는 것

- 각 기능의 입력 필드
- 각 기능의 출력 필드
- 허용하는 응답 길이
- 실패 시 반환 구조
- 출처 반환 형식
- 팀 문서 Q&A의 검색 범위

### 7.2 반드시 문서화할 것

- 기능별 입력 예시 형태
- 기능별 출력 JSON 형태
- 기능별 실패 사유 목록
- 팀 문서 Q&A의 문서 수집 대상
- 문서 청크 분리 기준
- 검색 결과를 몇 개까지 사용할지

### 7.3 STEP 1에서 처음 만드는 것

| 구분 | 먼저 만드는 항목 | 목적 |
| --- | --- | --- |
| 기능별 입출력 표 | 4개 기능의 입력 필드와 출력 필드 표 | 프론트와 백엔드가 어떤 데이터를 주고받는지 먼저 고정한다. |
| 프롬프트 초안 문서 | 기능별 역할, 입력, 출력, 금지사항 정리 | 기능 구현 전에 프롬프트 구조를 안정화한다. |
| 실패 처리 표 | 빈 응답, 형식 오류, 검색 실패 시 대응 기준 | 화면과 API에서 어떤 메시지를 보여줄지 맞춘다. |
| RAG 설계 메모 | 문서 수집 대상, 청크 기준, 검색 범위, 출처 구조 | 팀 문서 Q&A 설계를 초기에 고정한다. |
| 평가 기준 표 | 기능별로 어떤 응답을 좋은 결과로 볼지 정리 | 구현 후 품질 점검 기준을 만든다. |

STEP 1에서 중요한 것은 모델을 붙이는 것이 아니라, 입력과 출력이 흔들리지 않는 구조를 먼저 만드는 것이다.

## 8. STEP별 구현 가이드

### STEP 1

#### 목표

- 4개 AI 기능의 설계 기준을 고정한다.

#### 이 단계에서 해야 할 일

- 회의록 액션 아이템 추출의 입력과 출력 정의
- 문서 톤 변환의 입력과 출력 정의
- 진행 상황 요약의 입력과 출력 정의
- 팀 문서 Q&A의 입력과 출력 정의
- 기능별 실패 상황과 평가 기준을 같이 정리

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 프롬프트 설계 | 기능별로 어떤 역할을 하게 할지 먼저 정한다. 같은 모델을 쓰더라도 기능이 다르면 역할 설명이 달라야 한다. |
| Structured Output | 자유 문장 하나만 반환받지 말고, 프론트와 백엔드가 바로 처리할 수 있는 필드 구조를 먼저 정한다. |

#### 완료 기준

- 4개 기능 모두 입력값과 출력값을 설명할 수 있다.
- 성공 응답과 실패 응답 형태가 정리되어 있다.

### STEP 2

#### 목표

- 기능별 프롬프트 구조와 실패 처리 기준을 만든다.

#### 이 단계에서 해야 할 일

- 시스템 역할 문장 정리
- 사용자 입력이 어떤 필드로 들어오는지 정리
- 출력 형식 강제 기준 정리
- 실패 사유별 처리 기준 정리

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 프롬프트 구성 | 시스템 역할, 사용자 입력, 출력 형식, 금지사항을 분리한다. 기능 설명과 출력 형식 설명을 섞지 않는다. |
| 실패 처리 | 응답 누락, 형식 깨짐, 검색 결과 없음 같은 실패를 사전에 정의한다. 프론트와 백엔드가 어떤 메시지를 보여줄지도 함께 맞춘다. |

#### 완료 기준

- 기능마다 프롬프트 틀이 정리되어 있다.
- 실패 조건이 구현 전에 문서화되어 있다.

### STEP 3

#### 목표

- 액션 아이템 추출과 톤 변환 기능을 안정화한다.

#### 이 단계에서 해야 할 일

- 회의록 본문에서 어떤 문장을 할 일 후보로 볼지 기준 정리
- 담당자와 마감일은 확정값이 아니라 후보 또는 힌트로 다루는 기준 정리
- 톤 변환에서 허용할 톤 종류 고정
- 너무 장황하거나 원문 의도를 바꾸는 응답 방지 기준 정리

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 액션 아이템 추출 | 회의록 전체를 요약하는 것이 아니라, 실제로 실행해야 할 항목만 구조화해서 뽑는 방향으로 설계한다. |
| 톤 변환 | 공지 의미를 바꾸지 않고 표현만 바꾸는 기능으로 제한한다. 새로운 사실을 추가하지 않게 한다. |

#### 완료 기준

- 액션 아이템 추출 결과가 할 일 후보 형태로 안정적으로 나온다.
- 톤 변환 결과가 원문 의미를 크게 훼손하지 않는다.

### STEP 4

#### 목표

- 진행 상황 요약과 팀 문서 Q&A 설계를 구체화한다.

#### 이 단계에서 해야 할 일

- 진행 상황 요약에 포함할 데이터 범위 정리
- 팀 문서 Q&A의 문서 수집 대상 확정
- 문서 청크 기준 정리
- 임베딩과 검색 흐름 정리
- 출처 반환 구조 정리

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 진행 상황 요약 | 공지, 일정, 할 일, 회의록 중 무엇을 얼마나 반영할지 먼저 정한다. 모든 데이터를 무조건 넣지 말고 요약 기준을 정한다. |
| RAG 설계 | 문서 수집, 청크 분리, 임베딩, 질의 검색, 답변 생성, 출처 반환 단계를 분리해서 생각한다. 한 단계에 너무 많은 역할을 몰지 않는다. |

#### 완료 기준

- 진행 상황 요약 기준이 문서화되어 있다.
- 팀 문서 Q&A의 검색 흐름을 설명할 수 있다.

### STEP 5

#### 목표

- 4개 AI 기능을 실제 서비스 흐름에 연결한다.

#### 이 단계에서 해야 할 일

- 회의록 작성 화면에서 액션 아이템 추출 연결
- 공지 작성 화면에서 톤 변환 연결
- 대시보드에서 진행 상황 요약 연결
- 대시보드에서 팀 문서 Q&A 연결
- 출처 반환과 검색 범위 제한 점검

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 벡터 검색 | 팀 문서 Q&A에만 적용한다. 팀 범위를 벗어난 문서를 검색하지 않도록 팀 단위 필터를 같이 둔다. |
| 기능 통합 | 프론트에서 호출할 수 있는 응답 구조와 백엔드가 전달할 데이터 구조가 맞아야 한다. 기능별 입출력 필드를 다시 확인한다. |

#### 완료 기준

- 4개 기능이 실제 화면에서 호출된다.
- 팀 문서 Q&A는 답변과 출처를 함께 보여줄 수 있다.

### STEP 6

#### 목표

- 배포 환경에서 AI 기능 품질과 안정성을 검수한다.

#### 이 단계에서 해야 할 일

- 응답 길이 과다 여부 점검
- 의미 왜곡 여부 점검
- 출처 정확도 점검
- EC2 환경에서 모델 호출과 검색 동작 점검

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 응답 품질 점검 | 액션 아이템은 실행 가능해야 하고, 톤 변환은 의미를 유지해야 하고, 요약은 장황하지 않아야 하고, Q&A는 출처가 맞아야 한다. 기능별 평가 기준을 따로 둔다. |
| 운영 점검 | 로컬 테스트로 끝내지 말고 EC2에서 실제 요청 흐름을 확인한다. 환경 변수, 네트워크 연결, 검색 인덱스 상태를 같이 본다. |

#### 완료 기준

- 4개 기능이 모두 배포 환경에서 동작한다.
- 출처 반환과 실패 처리 흐름이 안정적이다.

## 9. 기능별 구현 체크 포인트

### 회의록에서 액션 아이템 추출

- 회의 내용 전체 요약과 구분되어야 한다.
- 할 일 후보, 담당자 힌트, 마감일 힌트가 구분되어야 한다.
- 없는 정보를 억지로 만들어내지 않도록 한다.

### 문서 톤 변환

- 원문 의미를 바꾸지 않아야 한다.
- 톤 종류는 문서에서 정한 범위만 허용한다.
- 결과가 너무 길어지지 않게 한다.

### 진행 상황 요약

- 단순 문장 합치기가 아니라 핵심 변화 중심으로 정리한다.
- 너무 많은 데이터가 들어와도 짧은 보고 형식으로 요약되게 한다.

### 팀 문서 Q&A

- 팀 범위를 벗어난 문서를 참조하지 않아야 한다.
- 답변에는 출처가 반드시 붙어야 한다.
- 검색 결과가 부족하면 모른다고 답하거나 범위를 알려주는 기준이 있어야 한다.

## 10. 최종 점검 질문

- 기능별 입력과 출력이 구현 전에 문서로 고정되어 있는가
- 프롬프트와 응답 형식이 기능마다 섞이지 않는가
- Q&A의 검색 범위와 출처 기준이 명확한가
- 프론트와 백엔드가 그대로 사용할 수 있는 구조인가

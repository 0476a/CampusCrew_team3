# BE-B STEP 6

## 이 STEP에서 왜 이걸 배우는가

마지막 단계에서는 새로운 API보다, 기존 API가 끝까지 잘 이어지는지가 더 중요하다.  
정상 흐름만 보지 말고, 잘못된 요청이나 빈 데이터도 같이 확인해야 한다.

## 이번 프로젝트에서 어디에 쓰는가

- 통합 테스트
- 예외 케이스 점검
- 대시보드와 CRUD 최종 확인
- 배포 전 기능 검증

## 반드시 알아야 하는 체크리스트

- [ ] 기능이 된다고 끝이 아니라 예외도 확인해야 한다는 것
- [ ] 데이터가 수정된 뒤 정합성이 유지돼야 한다는 것
- [ ] 데모에서 보여 줄 흐름을 따로 준비해야 한다는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 통합 테스트 | 기능들이 이어졌을 때 문제 없는지 보는 것 |  |
| 데이터 정합성 | 수정 후에도 데이터 관계가 맞는 상태 |  |
| 운영 오류 점검 | 배포 뒤에만 생길 수 있는 문제를 보는 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 핵심 테스트 흐름 | 공지, 일정, 할 일, 회의록 순서로 적는다 |  |
| 예외 케이스 | 잘못된 ID, 권한 없음, 빈 결과를 적는다 |  |
| 데모용 데이터 | 어떤 데이터가 꼭 있어야 하는지 적는다 |  |

## 막히면 이렇게 생각하면 된다

- 마지막일수록 기능 추가보다 검증이 중요하다.
- 정상 케이스만 테스트하면 실제 발표 때 흔들린다.
- 대시보드 수치와 상세 목록이 맞는지도 확인한다.
- 배포 1차 책임은 BE-A이고, BE-B는 리소스 기능 검증에 집중한다.

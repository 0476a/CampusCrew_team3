# BE-B STEP 2

## 이 STEP에서 왜 이걸 배우는가

기능을 빨리 만드는 것보다, 틀린 요청이 들어왔을 때 같은 방식으로 응답하는 것이 더 중요하다.  
성공 흐름만 먼저 보게 되기 쉬운데, 실제 협업에서는 실패 응답이 더 자주 부딪힌다.

## 이번 프로젝트에서 어디에 쓰는가

- 공통 에러 응답
- 요청값 검증
- DTO 설계

## 반드시 알아야 하는 체크리스트

- [ ] 입력값이 틀렸을 때 서버가 먼저 막아야 한다는 것
- [ ] 프론트가 이해할 수 있는 에러 형식이 필요하다는 것
- [ ] 엔터티와 응답 DTO는 다를 수 있다는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 예외 처리 | 잘못된 상황이 왔을 때 정해진 방식으로 응답하는 것 |  |
| DTO 검증 | 요청값이 말이 되는지 먼저 확인하는 것 |  |
| 공통 응답 구조 | 성공/실패의 모양을 비슷하게 맞추는 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 필수 입력값 규칙 | 제목, 날짜, 상태값 등 기본 규칙을 적는다 |  |
| 에러 응답 형태 | 코드, 메시지, 필드 에러 포함 여부를 적는다 |  |
| 프론트와 맞출 문구 | 너무 개발자스러운 문구는 피한다 |  |

## 막히면 이렇게 생각하면 된다

- 모든 에러를 하나의 메시지로 뭉개지 않는다.
- 날짜, enum, 빈 문자열 오류도 따로 생각한다.
- 프론트가 바로 쓰기 쉬운 구조인지 먼저 본다.

# Frontend STEP 1

## 이 STEP에서 왜 이걸 배우는가

이 단계에서는 아직 예쁘게 만드는 것이 핵심이 아니다.  
화면이 몇 개 있고, 어떤 순서로 이동하고, 어디까지가 공통 구조인지 먼저 잡아야 이후 STEP에서 덜 흔들린다.

## 이번 프로젝트에서 어디에 쓰는가

- 로그인 화면
- 회원가입 화면
- 팀 목록 화면
- 팀 내부 라우트 구조

## 반드시 알아야 하는 체크리스트

- [ ] 페이지와 공통 컴포넌트가 다르다는 것
- [ ] URL에 따라 다른 화면이 열린다는 것
- [ ] `teamId` 같은 값이 URL에 들어갈 수 있다는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 컴포넌트 분리 | 화면 전체를 담당하는 것과 여러 화면에서 같이 쓰는 조각을 나누는 것 |  |
| 라우팅 | 주소가 바뀌면 보여 주는 화면도 바뀌게 하는 것 |  |
| 상태 구분 | 입력값처럼 화면 안에서만 쓰는 값과 서버에서 받아오는 값을 나누는 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 화면 목록 | 지금 꼭 필요한 화면만 적는다 |  |
| URL 구조 | `/login`, `/signup`, `/teams`, `/teams/:teamId` 정도만 먼저 적는다 |  |
| 공통 레이아웃 | 여러 화면에서 같이 쓰일 영역이 무엇인지 적는다 |  |

## 막히면 이렇게 생각하면 된다

- 페이지 하나를 통째로 재사용하려고 하지 않는다.
- 로그인 전 화면과 로그인 후 화면을 먼저 나눈다.
- 팀 안쪽 화면은 `teamId` 기준으로 묶는다고 생각한다.

# Frontend STEP 1

## 이 STEP에서 왜 이걸 배우는가

프론트엔드는 보통 화면을 빨리 만들고 싶어지지만, 프로젝트 초반에는 "무슨 화면이 있고 어떤 순서로 이동하는지"를 먼저 이해해야 한다.  
이 STEP은 구현보다 먼저 프로젝트 전체 화면 구조를 읽는 단계다. React 문법 자체보다도 컴포넌트, 라우팅, 상태 구분이 무엇인지 감을 잡는 것이 더 중요하다.

## 이번 프로젝트에서 어디에 쓰는가

- 로그인 화면
- 회원가입 화면
- 팀 목록 화면
- 팀 내부 라우트 구조
- 공통 레이아웃

## 먼저 이해할 핵심 개념

- React 컴포넌트는 화면을 작은 단위로 나누어 만드는 방식이다.
- 라우팅은 URL에 따라 다른 화면을 보여 주는 규칙이다.
- 로컬 상태와 서버 상태는 성격이 다르다.
- 목록 화면, 상세 화면, 작성 화면은 역할이 다르다.
- 프로젝트 초반에는 예쁜 UI보다 화면 구조와 이동 흐름이 더 중요하다.

## 꼭 알아야 할 용어

| 용어 | 쉬운 설명 |
| --- | --- |
| 컴포넌트 | 화면을 구성하는 재사용 가능한 조각 |
| 페이지 | 라우트에 직접 연결되는 화면 단위 |
| 라우팅 | 주소에 따라 다른 페이지를 보여 주는 방식 |
| 레이아웃 | 여러 페이지에서 공통으로 쓰는 큰 틀 |
| 로컬 상태 | 입력창 값처럼 특정 화면 안에서만 쓰는 상태 |
| 서버 상태 | API를 다시 불러오고 동기화해야 하는 상태 |
| 파라미터 라우트 | `/teams/:teamId`처럼 URL 안에 값이 들어가는 라우트 |

## 추천 학습 순서

1. `PRD`, `UI_SPEC`, `SCREEN_FLOW`를 보고 어떤 화면이 있는지 먼저 본다.
2. React에서 컴포넌트가 무엇인지 이해한다.
3. React Router로 화면 이동을 어떻게 나누는지 본다.
4. 로컬 상태와 서버 상태가 왜 다른지 이해한다.
5. `FRONTEND_GUIDE`의 STEP 1을 다시 읽으며 프로젝트 구조에 대입한다.

## 자주 막히는 지점

- 페이지와 컴포넌트를 같은 것으로 생각한다.
- 로그인 전 화면과 로그인 후 화면을 섞어서 본다.
- `teamId` 같은 URL 파라미터가 왜 필요한지 감이 안 온다.
- 모든 상태를 한 곳에서 관리해야 한다고 생각한다.

## 가볍게 해볼 것

- `SCREEN_FLOW.md`를 보며 직접 페이지 이름만 적어 본다.
- `/teams/:teamId` 아래에 어떤 하위 화면이 있는지 적어 본다.
- "이건 페이지인지, 공통 컴포넌트인지"를 구분해서 메모해 본다.

## 핵심 질문

1. 로그인 이후 첫 화면이 왜 `/teams`인지 설명할 수 있는가
2. 페이지와 공통 컴포넌트의 차이를 예시와 함께 설명할 수 있는가
3. `/teams/:teamId` 같은 라우트가 왜 필요한지 설명할 수 있는가

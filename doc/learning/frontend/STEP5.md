# Frontend STEP 5

## 이 STEP에서 왜 이걸 배우는가

이 단계에서는 사용자가 직접 상태를 바꾸고 긴 텍스트를 다루게 된다.  
상태값, 회의록, AI 결과 표시가 섞이기 시작해서 화면이 복잡해지기 쉽다.

## 이번 프로젝트에서 어디에 쓰는가

- 할 일 목록과 상태 변경
- 회의록 작성
- AI 결과 표시

## 반드시 알아야 하는 체크리스트

- [ ] 상태값은 눈에 보이는 문구와 내부 값이 다를 수 있다는 것
- [ ] 회의록 본문과 AI 결과는 나눠서 보여 주는 편이 덜 헷갈린다는 것
- [ ] 상태가 바뀌면 목록도 다시 반영돼야 한다는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 상태 전이 | 할 일이 한 단계씩 바뀌는 흐름 |  |
| 리스트 갱신 | 데이터가 바뀐 뒤 화면도 다시 맞춰 주는 것 |  |
| AI 결과 표시 | AI가 준 내용을 사용자가 읽기 쉽게 보여 주는 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 상태값 문구 | 화면에서 어떤 한글 문구로 보여 줄지 적는다 |  |
| 회의록 입력 방식 | 긴 글 입력 시 필요한 최소 UX를 적는다 |  |
| AI 실패 시 문구 | 결과가 없을 때 어떤 문장을 보여 줄지 적는다 |  |

## 막히면 이렇게 생각하면 된다

- 상태 변경은 단순 스타일 변경이 아니라 데이터 변경이다.
- 회의록 원문과 추출 결과를 같은 박스에 섞지 않는다.
- AI 결과가 늦거나 실패해도 화면이 완전히 깨지면 안 된다.

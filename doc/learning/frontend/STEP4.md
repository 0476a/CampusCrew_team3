# Frontend STEP 4

## 이 STEP에서 왜 이걸 배우는가

이제부터는 "목록 보고, 작성하고, 다시 목록으로 돌아오는 흐름"이 계속 반복된다.  
화면은 만들더라도 저장 후 흐름이나 실패 상황을 빼먹기 쉬워서 이 부분을 먼저 잡아야 한다.

## 이번 프로젝트에서 어디에 쓰는가

- 공지 목록과 작성
- 일정 목록과 작성
- 저장 후 이동 흐름

## 반드시 알아야 하는 체크리스트

- [ ] 목록 화면과 작성 화면은 분리하는 편이 단순하다는 것
- [ ] 저장 중, 저장 성공, 저장 실패를 나눠야 한다는 것
- [ ] 날짜 입력은 보기보다 실수가 많다는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| CRUD | 만들기, 보기, 수정, 삭제의 기본 흐름 |  |
| 폼 상태 | 사용자가 입력하고 저장하는 동안 바뀌는 상태 |  |
| 날짜 검증 | 시작과 끝 시간이 말이 되는지 확인하는 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 저장 성공 후 이동 | 목록으로 갈지 상세로 갈지 적는다 |  |
| 날짜 포맷 | 백엔드와 같은 형식인지 확인한다 |  |
| 빈 상태 문구 | 데이터가 없을 때 보여 줄 문장을 적는다 |  |

## 막히면 이렇게 생각하면 된다

- 공지와 일정은 비슷해 보여도 날짜 처리에서 차이가 난다.
- 저장 버튼을 눌렀을 때 사용자가 지금 무슨 일이 일어나는지 알아야 한다.
- 실패 메시지는 개발자용이 아니라 사용자용이어야 한다.

# BE-A STEP 5

## 이 STEP에서 왜 이걸 배우는가

이 단계에서는 "수정할 수 있다"와 "수정해도 된다"를 구분해야 한다.  
상태 변경과 작성자 검증이 느슨하면 팀 데이터가 쉽게 꼬인다.

## 이번 프로젝트에서 어디에 쓰는가

- 할 일 상태 변경
- 회의록 작성자 검증
- 수정/삭제 권한 처리

## 반드시 알아야 하는 체크리스트

- [ ] 상태 변경은 단순 수정과 다를 수 있다는 것
- [ ] 작성자 확인이 필요한 기능이 있다는 것
- [ ] 권한 검증은 서비스 규칙에 가깝다는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 상태 변경 책임 | 누가 상태를 바꿀 수 있는지 정하는 규칙 |  |
| 작성자 검증 | 작성한 사람만 수정할 수 있게 확인하는 것 |  |
| 도메인 규칙 | 단순 저장이 아니라 서비스 안의 약속 같은 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 상태 변경 가능 사용자 | 팀원 전원인지 특정 역할인지 적는다 |  |
| 작성자 확인 기준 | 어떤 필드로 비교할지 적는다 |  |
| 실패 응답 | 권한 없음과 잘못된 상태를 나눠 적는다 |  |

## 실습 힌트

- 상태 전이 규칙을 표로 적으면 협업이 쉬워진다.
- 회의록 작성자 검증은 프론트 제어만으로 끝내지 않는다.

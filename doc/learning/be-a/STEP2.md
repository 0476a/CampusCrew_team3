# BE-A STEP 2

## 이 STEP에서 왜 이걸 배우는가

이제 로그인은 "화면 하나"가 아니라 전체 서비스 흐름의 시작점이 된다.  
JWT나 필터라는 단어에서 막히기 쉬운 단계이므로, 지금은 내부 구현을 완벽히 외우는 것보다 흐름을 이해하는 것이 먼저다.

## 이번 프로젝트에서 어디에 쓰는가

- 회원가입
- 로그인
- `me` 조회
- 보호 API 접근

## 반드시 알아야 하는 체크리스트

- [ ] 로그인 성공 시 토큰이 발급된다는 것
- [ ] 이후 요청에서 토큰을 함께 보낸다는 것
- [ ] 토큰이 없거나 잘못되면 보호 API를 막아야 한다는 것

## 먼저 읽을 문서

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 토큰 방식 인증 | 로그인 후 받은 값을 다음 요청마다 함께 보내는 방식 |  |
| 인증 필터 | 요청이 들어올 때 먼저 검사하는 문지기 같은 것 |  |
| 비밀번호 해시 | 비밀번호를 그대로 저장하지 않는 방식 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 로그인 성공 응답 | 토큰, 사용자 정보 등 필요한 값만 적는다 |  |
| 헤더 형식 | `Authorization` 헤더 규칙을 적는다 |  |
| 실패 상태 코드 | `401`, `403`을 언제 쓰는지 적는다 |  |

## 막히면 이렇게 생각하면 된다

- JWT 내부 원리를 모두 설명하지 못해도 흐름을 알면 된다.
- 비밀번호는 저장과 비교가 다르다는 점만 먼저 이해한다.
- 보호 API를 만들려면 "누가 요청했는지"를 꺼낼 수 있어야 한다.

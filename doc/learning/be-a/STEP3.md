# BE-A STEP 3

## 이 STEP에서 왜 이걸 배우는가

이 단계부터 팀 기능이 실제로 보이기 시작한다.  
팀 생성과 참여 흐름이 흔들리면 이후 공지, 일정, 할 일도 전부 흔들리기 때문에 기준을 먼저 맞춰야 한다.

## 이번 프로젝트에서 어디에 쓰는가

- 팀 생성
- 팀 목록
- 팀 상세
- 팀 참여

## 반드시 알아야 하는 체크리스트

- [ ] 사용자와 팀 사이에 참여 관계가 있다는 것
- [ ] 팀에 속했는지 확인해야 하는 기능이 많다는 것
- [ ] 참여 코드는 팀을 구분하는 보조 수단이라는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 도메인 모델링 | 팀, 사용자, 참여 관계를 코드 구조로 옮기는 것 |  |
| 권한 검증 | 이 사용자가 이 팀에 속하는지 확인하는 것 |  |
| 식별자 설계 | ID와 참여 코드가 각각 왜 필요한지 나누는 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 팀 생성 시 필요한 값 | 처음엔 최소 필드만 적는다 |  |
| 참여 성공/실패 기준 | 어떤 경우 실패인지 적는다 |  |
| 팀 응답 필드 | 프론트가 바로 쓸 필드를 적는다 |  |

## 실습 힌트

- 팀 참여 성공과 실패 케이스를 같이 적는다.
- 대시보드 진입에 필요한 팀 기본 정보를 정리한다.

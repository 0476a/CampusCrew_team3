# BE-A STEP 1

## 이 STEP에서 왜 이걸 배우는가

BE-A는 인증과 팀 도메인의 뼈대를 먼저 잡는 역할이다.  
처음 구조를 너무 복잡하게 잡으면 이후 STEP에서 계속 수정하게 되니, 지금은 "큰 흐름을 이해하는 것"이 더 중요하다.

## 이번 프로젝트에서 어디에 쓰는가

- 로그인 API
- 회원가입 API
- `me` 조회
- 팀 생성과 팀 참여의 기본 구조
- 최종 EC2 배포 책임

## 반드시 알아야 하는 체크리스트

- [ ] Controller, Service, Repository가 하는 일이 다르다는 것
- [ ] 요청과 응답은 JSON으로 주고받는다는 것
- [ ] 인증이 필요한 API와 아닌 API가 있다는 것

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 계층형 구조 | 역할마다 파일을 나눠서 관리하는 방식 |  |
| HTTP 요청/응답 | 프론트와 백엔드가 데이터를 주고받는 방식 |  |
| 인증 흐름 | 로그인 후 보호된 API를 쓸 수 있게 되는 흐름 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 공통 응답 구조 | 성공 응답에 무엇을 넣을지 적는다 |  |
| 인증 관련 용어 | 인증, 인가, 토큰 같은 말의 뜻을 적는다 |  |
| 팀 관련 기본 필드 | 팀 이름, 참여 코드처럼 먼저 필요한 것만 적는다 |  |

## 막히면 이렇게 생각하면 된다

- 처음부터 완벽한 구조보다 흐름이 맞는 구조가 더 중요하다.
- 엔터티를 그대로 프론트에 넘기지 않아도 된다.
- 인증과 팀 도메인의 책임을 대충 섞지 않는다.

# BE-A STEP 4

## 이 STEP에서 왜 이걸 배우는가

공지와 일정은 아무나 쓰면 안 된다.  
이 단계에서는 "기능이 되느냐"보다 "누가 할 수 있느냐"를 먼저 잡아야 이후 데이터가 꼬이지 않는다.

## 이번 프로젝트에서 어디에 쓰는가

- 공지 작성 권한
- 일정 작성 권한
- 팀 소속 검증

## 반드시 알아야 하는 체크리스트

- [ ] 같은 팀 멤버만 특정 작업을 하게 해야 한다는 것
- [ ] 권한 검증은 프론트만으로 끝나면 안 된다는 것
- [ ] 검증 실패도 일관된 응답이 필요하다는 것

## 먼저 읽을 문서

1. [BE_A_GUIDE](../../backend/BE_A_GUIDE.md)

## 쉬운 말로 정리

| 개념 | 쉬운 설명 | 내가 적어볼 내용 |
| --- | --- | --- |
| 권한 검증 | 이 사용자가 이 작업을 해도 되는지 확인하는 것 |  |
| 요청 검증 | 아예 잘못 들어온 값인지 확인하는 것 |  |
| 공통 로직 재사용 | 같은 검사를 여러 번 복붙하지 않게 묶는 것 |  |

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 팀 멤버 확인 위치 | 어디서 검사할지 적는다 |  |
| 권한 실패 응답 | 어떤 상태 코드와 메시지를 줄지 적는다 |  |
| BE-B와의 연결 규칙 | 팀 소속을 어떤 기준으로 넘길지 적는다 |  |

## 실습 힌트

- 팀 멤버가 아닌 사용자의 요청 흐름을 써 본다.
- 권한 관련 예외 메시지를 너무 상세히 줄지 고민해 본다.

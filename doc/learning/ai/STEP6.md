# AI STEP 6

## 이 STEP에서 왜 이걸 배우는가

마지막 단계에서는 AI가 똑똑해 보이는지보다, 발표와 운영에서 불안하지 않은지가 더 중요하다.  
성공 장면뿐 아니라 실패 장면까지 준비되어 있어야 실제 사용에서 덜 흔들린다.

## 이번 프로젝트에서 어디에 쓰는가

- 최종 AI 품질 점검
- 비용 통제
- 운영 fallback 준비

## 반드시 알아야 하는 체크리스트

- [ ] 모든 호출을 마음껏 하면 안 된다는 것
- [ ] 타임아웃과 실패 상황도 데모 일부라는 것
- [ ] 기능별 최소 성공 기준을 정해 두는 편이 좋다는 것

## 이번 STEP에서 바로 확인할 것

| 항목 | 힌트 | 작성란 |
| --- | --- | --- |
| 기능별 최소 성공 기준 | 어디까지 되면 통과로 볼지 적는다 |  |
| 호출 제한 기준 | 얼마나 자주 부를지 적는다 |  |
| 데모용 AI 시나리오 | 반드시 보여 줄 장면을 적는다 |  |

## 막히면 이렇게 생각하면 된다

- 비싼 기능일수록 더 적게, 더 안정적으로 부르는 편이 좋다.
- 실패 안내 문구도 사용자 경험 일부다.
- 최종 단계에서는 화려함보다 안정성이 중요하다.
- 배포 1차 책임은 BE-A이고, AI 담당은 기능 품질과 실패 응답 점검에 집중한다.

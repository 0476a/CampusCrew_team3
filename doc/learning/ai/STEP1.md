# AI STEP 1

## 이 STEP에서 왜 이걸 배우는가

AI 기능은 나중에 붙이는 옵션처럼 보이지만, 실제로는 입출력 구조를 초반에 제대로 이해하지 못하면 가장 늦게 흔들린다.  
모델 이름보다 먼저 "어떤 입력을 넣고 어떤 형식으로 결과를 받아야 안전한지"를 이해하는 것이 중요하다.

## 이번 프로젝트에서 어디에 쓰는가

- 액션 아이템 추출
- 톤 변환
- 진행 상황 요약
- 팀 문서 Q&A

## 먼저 이해할 핵심 개념

- AI 기능도 결국 입력과 출력 계약이 필요하다.
- 자유 문장보다 구조화된 응답이 연동에 유리하다.
- 프롬프트는 기능마다 역할이 달라야 한다.
- 실패 처리도 기능 설계의 일부다.
- 서비스 안의 AI는 "멋진 답변"보다 "안정적인 형식"이 더 중요하다.

## 꼭 알아야 할 용어

| 용어 | 쉬운 설명 |
| --- | --- |
| 프롬프트 | 모델에게 어떤 일을 시킬지 적는 입력 |
| 구조화 출력 | 일정한 JSON 형태처럼 고정된 응답 모양 |
| 스키마 | 응답에 어떤 필드가 있어야 하는지 정한 규칙 |
| fallback | 실패했을 때 대신 따르는 처리 방식 |
| 출처 반환 | 어떤 문서를 근거로 답했는지 같이 주는 것 |

## 추천 학습 순서

1. `AI_SPEC`에서 4개 기능의 입출력 구조를 읽는다.
2. 프롬프트와 구조화 출력 개념을 먼저 이해한다.
3. 실패 응답과 재시도 규칙이 왜 필요한지 본다.
4. `AI_GUIDE` STEP 1을 읽으며 기능별 차이를 본다.
5. STEP 2에서 실패 처리와 검증을 다룰 준비를 한다.

## 이번 STEP에서 바로 확인할 프로젝트 연결 포인트

| 항목 | 지금 확인할 것 |
| --- | --- |
| 기능 4개 | 기능별 입력과 출력이 어떻게 다른지 본다. |
| JSON 구조 | 프론트와 백엔드가 어떤 필드를 기대하는지 본다. |
| 실패 처리 | 실패 시 어떤 문구와 구조로 응답할지 본다. |
| 출처 | Q&A에서 왜 출처가 꼭 필요한지 본다. |

## 자주 막히는 지점

- 모델이 알아서 다 정리해 줄 것이라고 기대한다.
- 좋은 문장만 나오면 된다고 생각하고 출력 필드를 놓친다.
- 프롬프트와 응답 구조를 섞어서 생각한다.
- 실패 응답을 나중에 생각해도 된다고 여긴다.

## 가볍게 해볼 것

- 4개 기능 이름과 입력/출력 필드를 표처럼 적어 본다.
- "이 기능은 어떤 결과가 나오면 프론트가 바로 쓸 수 있을까?"를 적어 본다.
- 실패했을 때 사용자에게 어떤 문구가 보여야 할지 간단히 적어 본다.

## 핵심 질문

1. AI 기능도 왜 입력/출력 계약이 먼저 필요한가
2. 구조화 출력이 자유 문장보다 왜 연동에 유리한가
3. 실패 처리도 설계 범위라는 말을 설명할 수 있는가

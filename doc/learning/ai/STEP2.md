# AI STEP 2

## 이 STEP에서 왜 이걸 배우는가

AI는 성공했을 때보다 실패했을 때 서비스가 더 많이 흔들린다.  
이 단계에서는 모델 응답이 비어 있거나 형식이 깨지거나 출처가 부족할 때 어떻게 버틸지를 먼저 생각해야 한다.

## 이번 프로젝트에서 어디에 쓰는가

- AI 응답 파싱
- 실패 처리
- 응답 검증
- 재시도 기준
- 출처 검증

## 먼저 이해할 핵심 개념

- AI 응답은 항상 믿을 수 없다는 전제로 다뤄야 한다.
- JSON 구조가 맞는지 검증해야 한다.
- 일반 API 실패와 AI 실패는 구분하는 편이 좋다.
- 무조건 재시도하는 것이 정답은 아니다.
- 팀 문서 Q&A는 출처가 없으면 품질이 크게 떨어진다.

## 꼭 알아야 할 용어

| 용어 | 쉬운 설명 |
| --- | --- |
| 응답 파싱 | 문자열이나 JSON 응답에서 필요한 값을 꺼내는 과정 |
| 스키마 검증 | 응답이 약속한 구조를 지키는지 검사하는 것 |
| 재시도 | 실패했을 때 다시 한 번 요청하는 것 |
| hallucination | 근거 없이 그럴듯한 내용을 만들어내는 현상 |
| RAG | 관련 문서를 찾아 답변에 참고하는 방식 |

## 추천 학습 순서

1. `AI_SPEC`의 실패 처리 규칙을 읽는다.
2. 구조화 출력과 스키마 검증 개념을 다시 본다.
3. 응답 파싱과 재시도 기준을 정리한다.
4. RAG와 출처 반환이 왜 필요한지 이해한다.
5. `AI_GUIDE` STEP 2와 연결해서 프로젝트 기준으로 본다.

## 이번 STEP에서 바로 확인할 프로젝트 연결 포인트

| 항목 | 지금 확인할 것 |
| --- | --- |
| 빈 응답 처리 | 사용자에게 어떻게 보여 줄지 생각해 본다. |
| 형식 오류 처리 | JSON이 깨졌을 때 어떤 실패 응답을 줄지 본다. |
| 재시도 규칙 | 언제 재시도하고 언제 바로 실패 처리할지 생각해 본다. |
| 출처 검증 | Q&A에서 출처가 없으면 왜 문제인지 본다. |

## 자주 막히는 지점

- 실패를 숨기면 더 자연스럽다고 생각한다.
- 모든 실패를 무조건 재시도해야 한다고 생각한다.
- 응답 문자열만 있으면 된다고 보고 스키마 검증을 빼먹는다.
- RAG를 "문서를 많이 넣는 것" 정도로만 이해한다.

## 가볍게 해볼 것

- 빈 응답, 형식 오류, 출처 부족 상황을 각각 한 줄씩 적어 본다.
- "이 실패는 재시도할지, 바로 실패할지"를 나눠 적어 본다.
- Q&A 답변에 왜 출처 배열이 필요한지 설명해 본다.

## 핵심 질문

1. 프롬프트에서 역할, 입력, 출력, 금지사항을 왜 분리해야 하는가
2. 응답 형식 검증이 없으면 어떤 문제가 생기는가
3. 실패 사유를 구현 전에 문서화해야 하는 이유는 무엇인가

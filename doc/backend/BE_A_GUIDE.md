# CampusCrew 백엔드 BE-A 구현 가이드

## 1. 문서 목적

이 문서는 인증, 사용자, 팀, 팀 멤버십, 공통 보안과 예외 처리를 담당하는 백엔드 BE-A의 구현 지침서다.  
무엇을 만들어야 하는지는 `PRD.md`, `API_SPEC.md`, `ERD.md`, `PLAN.md`에 있고, 이 문서는 Spring에서 그 범위를 어떤 구조와 순서로 구현할지 정리한다.

## 3. BE-A 역할

BE-A는 서비스의 입구와 경계 조건을 담당한다.

- 회원가입, 로그인, 현재 사용자 조회
- JWT 기반 인증
- 보호 경로 설정
- 팀 생성, 팀 목록, 팀 상세, 팀 참여
- 팀 멤버십 검증
- 공통 예외 처리와 공통 응답 형식

즉, 다른 도메인이 안정적으로 동작하도록 기반을 만드는 역할이다.

## 4. 고정 스택과 사용 기준

| 항목 | 기술 | 사용 기준 |
| --- | --- | --- |
| 프레임워크 | Spring Boot | API 서버 기본 구조로 사용한다. |
| 웹 계층 | Spring Web | REST API를 구현한다. |
| 보안 | Spring Security | 인증, 인가, 보호 경로 관리를 담당한다. |
| 인증 방식 | JWT | 로그인 이후 요청 사용자를 식별하는 기준으로 사용한다. |
| 데이터 접근 | Spring Data JPA | 사용자, 팀, 팀 멤버십 관련 조회와 저장에 사용한다. |
| 검증 | Bean Validation | 요청 본문 검증 규칙을 통일한다. |

## 5. 구현 원칙

- 인증과 팀 도메인은 다른 도메인의 전제가 되므로 먼저 안정화한다.
- 예외 처리 형식은 초기에 고정하고 중간에 바꾸지 않는다.
- 권한 검증은 Controller가 아니라 Service 기준으로 둔다.
- 팀에 속한 사용자인지 확인하는 검증은 공통 흐름으로 만들어 재사용한다.
- 응답 필드는 프론트가 그대로 쓸 수 있게 DTO 기준으로 정리한다.
- 배포 환경에서 보안이 깨지지 않는지 항상 EC2 기준으로 확인한다.

## 6. 구현 시작 전 준비

### 6.1 먼저 정해야 하는 것

- 공개 경로와 보호 경로
- JWT 발급 방식과 전달 방식
- 인증 실패 응답 구조
- 사용자 식별 방식
- 팀 참여 코드 생성 및 검증 방식
- 공통 응답 형식과 공통 에러 응답 형식

### 6.2 패키지를 나누는 기준

- `auth`: 로그인, 회원가입, 현재 사용자
- `user`: 사용자 정보
- `team`: 팀 생성, 팀 조회, 팀 참여
- `membership`: 팀 소속 검증
- `global`: 공통 예외, 공통 응답, 보안 설정

핵심은 도메인과 공통 인프라를 섞지 않는 것이다.

### 6.3 STEP 1에서 처음 만드는 것

| 구분 | 먼저 만드는 항목 | 목적 |
| --- | --- | --- |
| 공통 설정 | 보안 설정 파일, 공통 예외 처리 위치, 공통 응답 구조 | 인증과 예외 처리 기준이 도중에 흔들리지 않게 한다. |
| 인증 도메인 뼈대 | 회원가입, 로그인, 현재 사용자 조회 흐름의 패키지 구조 | STEP 2에서 인증 API를 바로 구현할 수 있게 한다. |
| 팀 도메인 뼈대 | 팀 생성, 목록, 상세, 참여 흐름의 패키지 구조 | STEP 3 팀 기능 구현으로 자연스럽게 이어지게 한다. |
| 검증 기준 문서 | 공개 경로, 보호 경로, 필수 검증 항목 정리 | 프론트와 백엔드가 인증 조건을 같은 기준으로 이해하게 한다. |
| 응답 필드 합의 | 로그인 응답, 현재 사용자 응답, 팀 응답 필드 목록 | 프론트 연동 시 필드 이름 변경을 막는다. |

STEP 1의 핵심은 API를 많이 만드는 것이 아니라, 인증과 팀 도메인의 기반 규칙을 먼저 잠그는 것이다.

## 7. STEP별 구현 가이드

### STEP 1

#### 목표

- 프로젝트 구조와 보안 방향을 고정한다.

#### 이 단계에서 해야 할 일

- 인증 도메인과 팀 도메인 패키지 구조를 정한다.
- 공통 응답 형식과 에러 응답 형식을 문서 기준으로 정한다.
- 어떤 경로가 인증 없이 열리고 어떤 경로가 보호되는지 정리한다.
- 로그인 응답, 현재 사용자 응답, 팀 응답 필드 초안을 프론트와 맞춘다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| Spring Boot 패키지 구조 | Controller, Service, Repository 흐름이 한눈에 보이게 도메인별로 나눈다. 공통 예외와 보안 설정은 별도 `global` 성격의 위치에 둔다. |
| Spring Security 설계 | 로그인, 회원가입 같은 공개 경로와 팀 내부 기능처럼 보호해야 하는 경로를 먼저 문서 기준으로 정리한다. |
| 공통 응답 구조 | 성공 응답과 실패 응답의 큰 틀을 먼저 정한다. 그래야 이후 도메인 API가 같은 패턴으로 누적된다. |

#### 산출물

- 패키지 구조
- 공개/보호 경로 기준
- 공통 응답 기준

#### 완료 기준

- 새 API를 어디에 넣어야 하는지 바로 판단할 수 있다.
- 인증이 필요한 경로와 필요 없는 경로가 구분되어 있다.

### STEP 2

#### 목표

- 인증 흐름을 완성한다.

#### 이 단계에서 해야 할 일

- 회원가입 API를 만든다.
- 로그인 API를 만든다.
- 현재 사용자 조회 API를 만든다.
- JWT 발급과 검증 흐름을 붙인다.
- 인증 실패 응답을 공통화한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| Spring Security | 인증 필터 체계 안에서 사용자 식별이 어떻게 이뤄지는지 먼저 정리한다. 보호 API에서 인증 객체를 일관되게 꺼낼 수 있어야 한다. |
| JWT | 로그인 성공 시 토큰을 발급하고, 이후 요청마다 토큰을 검증해 사용자 정보를 복원하는 흐름으로 둔다. |
| Validation | 회원가입과 로그인 요청의 필수값과 길이 제한을 문서대로 검증한다. 잘못된 요청은 같은 형식의 에러 응답으로 반환한다. |

#### 산출물

- 회원가입 API
- 로그인 API
- 현재 사용자 조회 API
- JWT 인증 흐름

#### 협업 포인트

- 프론트와 토큰 저장 위치, 만료 시 처리 기준을 맞춘다.
- 로그인 응답 필드와 `me` 조회 응답 필드는 프론트가 그대로 사용할 수 있어야 한다.

#### 완료 기준

- 토큰 없이 보호 API 접근 시 차단된다.
- 로그인 성공 후 현재 사용자 조회가 가능하다.
- 잘못된 로그인 요청에 대한 응답 형식이 일정하다.

### STEP 3

#### 목표

- 팀 도메인을 완성한다.

#### 이 단계에서 해야 할 일

- 팀 생성 API를 만든다.
- 내가 속한 팀 목록 조회 API를 만든다.
- 팀 상세 조회 API를 만든다.
- 참여 코드 기반 팀 참여 API를 만든다.
- 팀 멤버십 검증 흐름을 공통화한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| Spring Data JPA | 사용자와 팀, 팀 멤버십 관계를 기준으로 저장과 조회를 만든다. 팀 생성 시 팀장 멤버십이 함께 생성되도록 흐름을 맞춘다. |
| 멤버십 검증 | 팀 하위 기능은 모두 팀 소속 여부가 필요하므로, 재사용 가능한 검증 로직을 만든다. |
| DTO 설계 | 팀 목록과 팀 상세 응답에는 프론트가 바로 표시할 수 있는 필드만 넣고, 내부 엔터티 구조를 그대로 노출하지 않는다. |

#### 산출물

- 팀 생성 API
- 팀 목록 API
- 팀 상세 API
- 팀 참여 API
- 팀 소속 검증 흐름

#### 완료 기준

- 사용자 기준으로 팀 목록을 안정적으로 조회할 수 있다.
- 참여 코드를 통해 팀에 들어갈 수 있다.
- 이후 도메인에서 팀 멤버십 검증을 재사용할 수 있다.

### STEP 4

#### 목표

- 공지와 일정 도메인이 사용할 권한과 검증 기반을 완성한다.

#### 이 단계에서 해야 할 일

- 팀 소속 사용자만 팀 내부 리소스에 접근하게 만든다.
- 팀 ID 검증, 작성자 검증 같은 공통 로직을 정리한다.
- 존재하지 않는 팀, 권한 없음, 잘못된 입력에 대한 예외 처리를 고정한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| Service 계층 검증 | 권한과 소속 검증은 요청을 받은 직후 Controller에서 끝내지 말고, 실제 비즈니스 로직이 있는 Service에서 보장한다. |
| 공통 예외 처리 | 예외마다 응답 구조가 달라지지 않게 전역 처리 기준으로 묶는다. |
| Spring Security + 멤버십 검증 | 인증된 사용자라는 사실과 팀 소속 사용자라는 사실은 별개다. 두 조건을 분리해서 검증한다. |

#### 완료 기준

- 공지와 일정 API가 권한 기준 없이 열리지 않는다.
- 팀이 없거나 권한이 없을 때 응답이 일관된다.

### STEP 5

#### 목표

- 할 일 상태 변경과 회의록 작성자 규칙을 안정화한다.

#### 이 단계에서 해야 할 일

- 할 일 상태 변경 권한 규칙을 구현한다.
- 회의록 수정과 삭제의 작성자 규칙을 구현한다.
- 팀 ID 기준으로 할 일과 회의록이 일관되게 동작하는지 검증한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 상태값 검증 | 문서에 정의된 상태값만 허용하고, 임의 문자열은 차단한다. |
| 작성자 검증 | 회의록은 누가 썼는지 기준이 중요하므로 수정과 삭제 권한을 명확히 둔다. |
| 통합 검증 | 팀 ID, 사용자 ID, 작성자 ID가 서로 다른 방향으로 흩어지지 않는지 확인한다. |

#### 완료 기준

- 할 일 상태 변경 규칙이 명확하다.
- 회의록 수정과 삭제 권한이 안정적으로 동작한다.

### STEP 6

#### 목표

- 배포 환경에서 인증과 팀 기능을 최종 검수한다.

#### 이 단계에서 해야 할 일

- JWT 키와 환경 변수를 정리한다.
- EC2에서 보호 경로가 정상적으로 동작하는지 확인한다.
- 프론트가 쓰는 필드와 실제 응답 필드가 끝까지 같은지 점검한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 환경 변수 | 로컬 값과 배포 값을 분리하고, 민감한 값은 소스 코드에 두지 않는다. |
| 보안 점검 | 로컬에서만 테스트하지 말고 EC2에서 직접 인증 흐름을 확인한다. |
| 응답 검수 | 프론트 화면이 실제로 참조하는 응답 구조를 기준으로 마지막 점검을 한다. |

#### 완료 기준

- EC2에서 로그인부터 팀 진입까지 문제 없이 동작한다.
- 인증 관련 오류가 프론트에서 예측 가능한 형태로 보인다.

## 8. 최종 점검 질문

- 공개 경로와 보호 경로가 섞여 있지 않은가
- 멤버십 검증이 빠진 팀 하위 API는 없는가
- 예외 응답 형식이 API마다 달라지지 않는가
- JWT 검증 실패 시 흐름이 일관적인가

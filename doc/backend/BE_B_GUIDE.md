# CampusCrew 백엔드 BE-B 구현 가이드

## 1. 문서 목적

이 문서는 공지, 일정, 할 일, 회의록, 대시보드 집계를 담당하는 백엔드 BE-B의 구현 지침서다.  
무엇을 구현해야 하는지는 `PRD.md`, `API_SPEC.md`, `ERD.md`에 있고, 이 문서는 그 도메인을 Spring에서 어떤 순서와 방식으로 구현할지 정리한다.

## 3. BE-B 역할

BE-B는 팀 활동 데이터 자체를 책임진다.

- 공지 CRUD
- 일정 CRUD
- 할 일 CRUD
- 회의록 CRUD
- 팀 대시보드 집계
- AI 기능이 사용할 문서 데이터 정리

즉, 사용자가 실제로 매일 보는 협업 데이터의 중심을 만든다.

## 4. 고정 스택과 사용 기준

| 항목 | 기술 | 사용 기준 |
| --- | --- | --- |
| 프레임워크 | Spring Boot | 도메인별 API 서버 구조를 만든다. |
| 웹 계층 | Spring Web | 공지, 일정, 할 일, 회의록, 대시보드 API를 구현한다. |
| 데이터 접근 | Spring Data JPA | CRUD와 정렬, 최신순, 마감 임박 조회를 구현한다. |
| 검증 | Bean Validation | 요청 본문의 길이, 필수값, 날짜 규칙을 검증한다. |
| DTO | 수동 DTO 설계 | 프론트와 AI가 쓰는 응답 구조를 명확히 분리한다. |

## 5. 구현 원칙

- 엔터티는 ERD 기준 이름과 관계를 유지한다.
- CRUD 흐름은 도메인이 달라도 같은 패턴으로 맞춘다.
- 목록, 상세, 생성, 수정, 삭제가 뒤섞이지 않게 메서드 책임을 나눈다.
- 날짜와 상태값 검증은 요청 단계와 서비스 단계 모두에서 점검한다.
- 대시보드 집계는 여러 API를 프론트에서 억지로 합치지 않도록 별도 응답으로 만든다.
- AI가 쓸 데이터는 사람이 읽기 쉬운 것보다 구조적으로 안정적인 것이 더 중요하다.

## 6. 구현 시작 전 준비

### 6.1 먼저 정해야 하는 것

- 공지, 일정, 할 일, 회의록 엔터티 필드
- 작성자와 팀 관계 처리 방식
- 상태값 종류와 허용 범위
- 날짜와 시간 검증 기준
- 대시보드 집계 응답 구조
- AI 기능이 재사용할 데이터 범위

### 6.2 패키지를 나누는 기준

- `notice`
- `schedule`
- `task`
- `meetingnote`
- `dashboard`
- `global` 또는 공통 DTO/예외 처리 위치

### 6.3 STEP 1에서 처음 만드는 것

| 구분 | 먼저 만드는 항목 | 목적 |
| --- | --- | --- |
| 도메인 뼈대 | 공지, 일정, 할 일, 회의록, 대시보드 패키지 구조 | STEP별 구현 범위를 도메인 단위로 쌓기 쉽게 한다. |
| 엔터티 초안 | ERD 기준 엔터티와 연관관계 정리 | DTO와 조회 API를 만들 때 기준점으로 쓴다. |
| 요청/응답 초안 | 공지, 일정, 할 일, 회의록의 생성 요청과 목록 응답 필드 정리 | 프론트가 어느 필드를 기다리는지 먼저 맞춘다. |
| 집계 기준 문서 | 대시보드에서 어떤 데이터를 몇 개씩 묶어줄지 정리 | STEP 3 이후 조회 API 방향을 흔들리지 않게 한다. |
| AI 연계 메모 | AI가 어떤 문서 데이터를 가져갈지 메모 | STEP 4와 STEP 5에서 응답 구조를 다시 뜯어고치지 않게 한다. |

STEP 1에서 먼저 해야 하는 것은 CRUD 구현이 아니라, 각 도메인의 데이터 형태와 조회 단위를 잠그는 일이다.

## 7. STEP별 구현 가이드

### STEP 1

#### 목표

- 프로젝트 시작 전에 도메인 구조와 데이터 관계를 이해한다.
- ERD와 API 명세를 읽고 리소스 중심 사고를 익힌다.
- STEP 2에서 DTO와 검증을 만들기 전에 필요한 데이터 감각을 준비한다.

#### 이 단계에서 해야 할 일

- 공지, 일정, 할 일, 회의록, 대시보드 도메인이 어떻게 나뉘는지 읽는다.
- `ERD.md` 기준으로 엔터티 관계를 다시 확인한다.
- 어떤 도메인이 작성자 정보를 갖고, 어떤 도메인이 상태값을 갖는지 이해한다.
- 대시보드 집계에 필요한 조회 단위를 적어 본다.
- STEP 2로 넘어가기 전에 CRUD, 엔터티 관계, DTO 분리 감각을 준비한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| Spring Boot 패키지 구조 | 도메인별로 Controller, Service, Repository를 나누어 책임을 분리한다. |
| ERD 연계 | 엔터티 이름, 컬럼 이름, 연관관계를 문서 기준으로 맞춘다. 이후 필드명을 임의로 줄이거나 바꾸지 않는다. |

#### 완료 기준

- 각 도메인이 어디까지 책임지는지 설명할 수 있다.
- 엔터티 관계를 읽고 따라갈 수 있다.
- STEP 2에서 요청/응답 구조를 만들 준비가 되어 있다.

### STEP 2

#### 목표

- 각 도메인의 요청과 응답 구조를 먼저 고정한다.

#### 이 단계에서 해야 할 일

- 공지, 일정, 할 일, 회의록의 생성 요청 DTO를 만든다.
- 목록과 상세 조회 응답 DTO를 만든다.
- 잘못된 입력과 없는 리소스에 대한 예외 형식을 BE-A와 맞춘다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| DTO 설계 | 엔터티를 직접 응답으로 내보내지 않고, 프론트가 필요한 필드만 담는 구조로 정리한다. |
| Validation | 제목, 본문, 날짜, 상태값, 설명 길이 같은 검증을 요청 DTO에 먼저 둔다. |
| 공통 예외 처리 | 없는 공지, 없는 일정, 잘못된 상태값처럼 반복되는 오류는 같은 방식으로 응답한다. |

#### 완료 기준

- 도메인별 입력과 출력 구조가 명확하다.
- 프론트가 어떤 필드를 받는지 API 문서와 맞는다.

### STEP 3

#### 목표

- 팀 대시보드가 먼저 동작할 수 있도록 읽기 API를 만든다.

#### 이 단계에서 해야 할 일

- 최근 공지 조회
- 다가오는 일정 조회
- 내 할 일 조회
- 최근 회의록 조회

대시보드에 필요한 읽기 전용 API를 먼저 만든다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| JPA 조회 메서드 | 최신순, 시작일 기준, 마감 임박 순서처럼 조회 기준이 분명한 정렬부터 만든다. |
| 대시보드 보조 조회 | 프론트가 여러 화면용 API를 억지로 재사용하지 않도록 대시보드에 필요한 데이터만 모은 읽기 흐름을 만든다. |

#### 완료 기준

- 대시보드에 들어갈 데이터가 개별 조회로 준비된다.
- 프론트가 카드 단위로 바로 붙일 수 있다.

### STEP 4

#### 목표

- 공지와 일정 도메인을 완성한다.
- AI 기능 중 진행 상황 요약과 Q&A가 쓸 문서 기반을 준비한다.

#### 이 단계에서 해야 할 일

- 공지 생성, 목록, 상세, 수정, 삭제
- 일정 생성, 목록, 상세, 수정, 삭제
- 날짜 검증 규칙 정리
- AI가 읽을 수 있는 공지와 일정 데이터 구조 점검

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| CRUD 패턴 | 생성, 목록, 상세, 수정, 삭제의 메서드 이름과 책임을 통일한다. |
| 날짜 처리 | 일정 시작 시간과 종료 시간 관계를 명확히 검증한다. 단순히 값이 있는지만 보지 말고 논리 순서를 확인한다. |
| AI 데이터 준비 | 진행 상황 요약과 Q&A는 문서 내용을 다시 활용하므로, 공지와 일정 응답에 핵심 텍스트와 날짜가 빠지지 않게 한다. RAG 원본은 `NOTICE`, `SCHEDULE` 같은 문서 타입 기준으로 식별된다는 점을 같이 맞춘다. |

#### 완료 기준

- 공지와 일정 전체 CRUD가 동작한다.
- 진행 상황 요약과 Q&A에 필요한 공지/일정 데이터가 안정적으로 나온다.

### STEP 5

#### 목표

- 할 일과 회의록 도메인을 완성한다.
- 대시보드 집계를 마무리한다.
- AI가 쓸 핵심 데이터 구조를 정리한다.

#### 이 단계에서 해야 할 일

- 할 일 생성, 목록, 상세, 수정, 삭제, 상태 변경
- 회의록 생성, 목록, 상세, 수정, 삭제
- 팀 대시보드 집계 응답
- 액션 아이템 추출과 Q&A에 필요한 회의록/할 일 데이터 정리

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| CRUD 구현 | 할 일과 회의록도 공지/일정과 같은 패턴으로 만든다. 그래야 프론트와 백엔드 둘 다 예측 가능해진다. |
| 상태값 응답 | 할 일 상태는 프론트가 그대로 표시할 수 있게 안정적인 문자열로 반환한다. |
| 집계 응답 | 대시보드는 여러 리소스를 한 번에 보여주므로 별도 집계 응답을 두는 편이 낫다. |
| AI 데이터 준비 | 액션 아이템 추출은 회의록 본문이 중요하고, Q&A는 여러 문서 조합이 중요하다. 어떤 필드를 AI에 넘길지 의식하면서 응답을 정리하고, 원본 문서 수정/삭제 시 RAG 데이터가 함께 재생성·정리되는 흐름을 염두에 둔다. |

#### 완료 기준

- 할 일과 회의록 전체 CRUD가 동작한다.
- 대시보드 집계 응답이 프론트 연결 기준으로 안정적이다.
- AI가 읽을 문서 데이터가 빠지지 않는다.

### STEP 6

#### 목표

- 배포 환경에서 리소스 API와 집계 API를 최종 점검한다.

#### 이 단계에서 해야 할 일

- 프론트와 AI가 쓰는 응답 구조 최종 확인
- 조회 성능 과도 저하 여부 점검
- 배포본 기준 리소스 API 점검

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 응답 검수 | 프론트가 실제로 읽는 필드와 AI가 실제로 읽는 필드를 기준으로 마지막 확인을 한다. |
| 배포 환경 점검 | 로컬에서 빠르더라도 실제 배포 환경과 AI 서버 연동 환경에서 지나치게 느린 조회가 없는지 확인한다. |

#### 완료 기준

- 프론트와 AI가 모두 사용할 수 있는 응답 구조가 확정된다.
- 주요 CRUD와 대시보드 조회가 배포 환경에서도 안정적이다.

## 8. 최종 점검 질문

- 공지, 일정, 할 일, 회의록의 CRUD 패턴이 서로 일관적인가
- 상태값과 날짜 검증이 요청 단계와 서비스 단계에서 모두 보장되는가
- 대시보드 집계 응답이 프론트 연결 기준으로 충분한가
- AI가 필요한 문서 데이터를 누락 없이 받을 수 있는가

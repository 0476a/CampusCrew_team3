# CampusCrew 화면 명세서

## 1. 문서 목적
이 문서는 프론트엔드 구현 기준을 빠르게 맞추기 위한 화면 명세서다.  
디자인 시안이 없는 상태에서도 화면 목적, 핵심 UI, 입력값, API 연동 지점을 바로 파악할 수 있도록 작성했다.

참고용 화면 시안은 [Figma Make UI 시안](https://www.figma.com/make/22ywy6HuSpF6N6cJC56eQ1/CampusCrew-UI-Design?fullscreen=1&t=n6QahEAS9SYPyqsq-1&preview-route=%2Fteam%2F1)에서 본다.

## 2. 공통 화면 원칙
- 로그인 이후 기본 레이아웃은 좌측 네비게이션 영역 + 우측 콘텐츠 영역 구조를 사용한다.
- 모든 목록 화면은 `로딩 상태`, `빈 상태`, `에러 상태`를 가진다.
- 모든 작성/수정 화면은 저장 성공 후 목록 화면으로 이동한다.
- 모든 작성 기능은 별도 화면 방식으로 구현한다.

## 3. 플랫폼 적용 원칙

- 이 프로젝트는 웹 기준으로 본다.
- 각 화면은 React Router 기반 페이지로 구현한다.
- 저장 후 이동, 보호 라우트, API 연동 규칙은 이 문서와 `API_SPEC.md`를 따른다.

## 4. UI 제약사항

- 로그인 이후 기본 구조는 `사이드바 + 콘텐츠 영역`으로 맞춘다.
- 목록 화면은 `로딩`, `빈 상태`, `에러 상태`를 가진다.
- 작성과 수정은 별도 화면으로 분리한다.
- 폼은 복잡하게 나누지 말고 읽기 쉬운 단일 컬럼을 우선한다.
- 공지, 일정, 할 일, 회의록은 같은 서비스 안의 화면처럼 일관된 레이아웃과 버튼 스타일을 유지한다.
- 상태는 색상만이 아니라 텍스트나 배지로 함께 보여 준다.
- 팀 내부 메뉴 순서는 `대시보드 > 공지 > 일정 > 할 일 > 회의록 > 팀 정보`를 기준으로 한다.
- 반응형은 "모바일 폭에서도 무너지지 않는가"를 우선으로 본다.

## 5. 입력 제약사항

| 항목 | 규칙 |
| --- | --- |
| 이메일 | 이메일 형식을 만족해야 한다. 최대 길이는 `100자`다. |
| 비밀번호 | 최소 `8자`, 최대 `20자`로 제한한다. |
| 이름 | 최소 `2자`, 최대 `20자`로 제한한다. |
| 학번 | 숫자만 입력한다. 길이는 `8-10자`로 제한한다. |
| 팀명 | 최소 `2자`, 최대 `30자`로 제한한다. |
| 과목명 | 최소 `2자`, 최대 `30자`로 제한한다. |
| 참여 코드 | 영문 대문자와 숫자만 허용한다. 길이는 `6자`로 고정한다. |
| 제목 | 최소 `2자`, 최대 `50자`로 제한한다. |
| 본문 | 최대 `2000자`로 제한한다. |
| 설명 | 최대 `500자`로 제한한다. |
| 장소 | 최대 `50자`로 제한한다. |
| 날짜 | 시작 일시는 종료 일시보다 늦을 수 없다. |
| 마감일 | 오늘 이전 날짜를 선택할 수 없다. |
| 상태값 | `TODO`, `IN_PROGRESS`, `DONE` 외의 값은 허용하지 않는다. |
| 공백 처리 | 입력값 앞뒤 공백은 저장 전에 제거한다. |
| 필수값 | 필수 입력값이 비어 있으면 저장 버튼을 비활성화한다. |

## 5.1 AI 적용 규칙

- 회의록 작성 화면에는 `액션 아이템 추출` 버튼을 둔다.
- 공지 작성 화면에는 `톤 변환` 버튼을 둔다.
- 팀 대시보드에는 `진행 상황 요약` 버튼과 `팀 문서 Q&A` 영역을 둔다.

## 7. 공통 UI 요소 명세

### 7.1 AppLayout
- 목적: 인증 이후 공통 화면 구조 제공
- 포함 요소: 네비게이션 영역, 상단 제목 영역, 콘텐츠 영역

### 7.2 TeamSidebar
- 목적: 팀 내부 메뉴 이동
- 메뉴: 대시보드, 공지, 일정, 할 일, 회의록, 팀 정보

### 7.3 PageHeader
- 목적: 화면 제목, 설명, 주요 액션 버튼 표시
- 제목은 현재 화면의 기능명을 그대로 사용한다.
- 액션 버튼 문구는 생성 화면에서는 `새로 만들기`, 수정 화면에서는 `수정하기`로 통일한다.

### 7.4 EmptyState
- 목적: 데이터가 없을 때 안내 문구와 CTA 제공
- 안내 문구는 `등록된 데이터가 없습니다.` 형식으로 통일한다.
- CTA 문구는 `새로 만들기`로 통일한다.

### 7.5 StatusBadge
- 목적: 할 일 상태, 중요 태그 등을 시각적으로 표시
- 상태값: `TODO`, `IN_PROGRESS`, `DONE`

## 8. 상세 화면 명세

## 8.1 AUTH-01 로그인

### 목적
- 기존 사용자가 서비스에 로그인한다.

### 주요 UI
- 이메일 입력
- 비밀번호 입력
- 로그인 버튼
- 회원가입 이동 링크

### 유효성 검사
- 이메일 형식 검사
- 이메일 최대 길이 `100자`
- 비밀번호 최소 `8자`, 최대 `20자`
- 필수값 누락 시 제출 금지

### 성공 시 동작
- Access Token 저장
- `/teams`로 이동

### 실패 시 메시지
- "이메일 또는 비밀번호가 올바르지 않습니다."

### 연동 API
- `POST /api/auth/login`

## 8.2 AUTH-02 회원가입

### 주요 UI
- 이름
- 학번
- 이메일
- 비밀번호
- 회원가입 버튼

### 유효성 검사
- 모든 필수값 검사
- 이메일 형식 검사
- 이름 최소 `2자`, 최대 `20자`
- 학번 숫자만 허용, 길이 `8-10자`
- 비밀번호 최소 `8자`, 최대 `20자`

### 연동 API
- `POST /api/auth/signup`

## 10. 화면 검수 체크리스트
- 로그인 후 잘못된 화면 접근 시 보호 라우트 또는 보호 화면 전환이 동작하는가
- 팀 ID가 없는 상태에서 팀 하위 화면 접근을 막는가
- 목록 데이터가 없을 때 빈 상태가 자연스러운가
- 저장 성공/실패 메시지가 명확한가
- 할 일 상태와 일정 날짜가 한눈에 읽히는가
- 모바일 폭에서 화면이 깨지지 않는가
- 작성 화면과 목록 화면의 역할이 섞이지 않는가
- 공통 버튼, 입력창, 카드 스타일이 크게 어긋나지 않는가

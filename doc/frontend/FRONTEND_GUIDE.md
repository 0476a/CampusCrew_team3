## 3. 프론트엔드 역할

프론트엔드 담당자는 화면을 그리는 역할만 하는 것이 아니다. 아래 항목을 모두 책임진다.

- 라우트 구조 설계
- 화면 전환 흐름 정리
- 서버 데이터 표시 방식 정리
- 입력 폼 구조 통일
- 로딩, 빈 상태, 오류 상태 처리
- AI 기능이 붙는 위치와 사용자 경험 정리
- 배포 후 실제 브라우저 동작 확인

## 4. 고정 스택과 사용 기준

| 항목 | 기술 | 사용 기준 |
| --- | --- | --- |
| 프레임워크 | React | 화면 단위를 컴포넌트로 나누고, 상태와 화면을 연결하는 기본 구조로 사용한다. |
| 라우팅 | React Router | 로그인 이후 화면 구조와 팀 공간 내부 하위 화면 전환을 관리한다. |
| 서버 상태 | TanStack Query | 목록 조회, 상세 조회, 생성, 수정, 삭제, AI 요청 결과를 서버 상태 기준으로 관리한다. |
| 스타일 | CSS Modules | 화면별 스타일 파일과 공통 레이아웃 스타일을 분리해 관리한다. |
| API 통신 | fetch 또는 팀이 정한 단일 HTTP 클라이언트 | 프로젝트 전체에서 하나로 통일하고, 인증 헤더 처리 방식도 한 곳에서 맞춘다. |

## 5. 프론트엔드 구현 원칙

- 화면 이름, 입력 이름, 상태값 이름은 문서 기준 이름을 그대로 사용한다.
- 하나의 화면은 하나의 목적만 가지게 만든다.
- 작성 기능은 목록 화면 안에 억지로 넣지 말고 별도 작성 화면으로 분리한다.
- 서버 데이터는 TanStack Query로 관리하고, 입력 중인 폼 값은 로컬 상태로 관리한다.
- 생성, 수정, 삭제 이후에는 화면을 억지로 수동 수정하지 말고 재조회 기준으로 상태를 맞춘다.
- 화면이 완성되더라도 로딩 상태, 빈 상태, 오류 상태가 없으면 완료로 보지 않는다.
- AI 기능은 독립 서비스가 아니라 기존 화면 안에서 사용하는 도구로 배치한다.

## 6. 구현 시작 전 준비

### 6.1 먼저 정해야 하는 것

- 라우트 목록
- 공통 레이아웃 구조
- 인증 전 화면과 인증 후 화면 구분
- API 호출 파일 위치
- 공통 입력 컴포넌트와 버튼 컴포넌트 범위
- 에러 메시지 표기 위치

### 6.2 폴더를 나누는 기준

- `pages`: 라우트에 직접 연결되는 화면
- `components`: 공통 UI
- `features`: 도메인별 화면 조각과 폼
- `services` 또는 `api`: HTTP 요청
- `hooks`: 쿼리 훅, 인증 훅, 공통 처리 훅
- `styles`: 공통 스타일

핵심은 화면 단위와 공통 요소를 섞지 않는 것이다.

## 7. STEP별 구현 가이드

### STEP 1

#### 목표

- 프로젝트 시작 전에 전체 화면 흐름을 이해한다.
- React 프로젝트 구조를 읽고 어떤 식으로 시작할지 감을 잡는다.
- 이후 STEP에서 재사용할 공통 레이아웃과 라우트 감각을 만든다.

#### 이 단계에서 해야 할 일

- 로그인, 회원가입, 팀 목록, 팀 생성, 팀 대시보드, 공지, 일정, 할 일, 회의록, 팀 정보 화면이 어떤 순서로 이어지는지 읽는다.
- 팀 공간 내부에서 공통으로 쓰는 레이아웃이 왜 필요한지 이해한다.
- 작성 화면과 목록 화면을 왜 분리하는지 파악한다.
- `UI_SPEC.md`의 화면 구성을 실제 컴포넌트 단위로 나눠보는 연습을 한다.
- 화면별로 연결될 API와 필요한 상태를 가볍게 표시해 본다.
- STEP 2로 넘어가기 전에 React Router, 상태 구분, 공통 컴포넌트 감각을 준비한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| React Router | 먼저 URL 구조를 고정한다. 인증 전 영역과 인증 후 영역을 나누고, 팀 ID가 필요한 화면은 팀 공간 하위 라우트로 묶는다. |
| CSS Modules | 페이지 전용 스타일과 공통 레이아웃 스타일을 분리한다. 페이지마다 고유한 스타일 파일을 두고, 공통 레이아웃 스타일은 별도 모듈로 뺀다. |
| 컴포넌트 구조 | 화면 컴포넌트는 데이터 흐름과 화면 구성을 담당하고, 공통 컴포넌트는 입력창, 버튼, 카드, 리스트, 배지처럼 반복 요소만 담당하게 둔다. |

#### 산출물

- 화면 흐름 이해
- 라우트 구조 메모
- 공통 레이아웃과 화면 분리 감각

#### 이 단계에서 흔히 생기는 문제

- 목록 화면과 작성 화면을 하나로 합치려다 구조가 복잡해짐
- 팀 공간 내부 라우트를 너무 늦게 정해서 이후 수정 비용이 커짐
- 공통 컴포넌트와 화면 전용 컴포넌트를 섞어 버림

#### 완료 기준

- 로그인 이후 어느 화면으로 이동하는지 설명할 수 있다.
- 팀 공간 하위 화면 구조를 읽고 따라갈 수 있다.
- STEP 2에서 화면 구현을 시작할 준비가 되어 있다.

### STEP 2

#### 목표

- 인증 흐름을 완성한다.
- 로그인 전과 로그인 후 화면 접근 규칙을 분리한다.
- 서버 초기 셋팅이 진행되는 동안 프론트는 로컬 통신 기준으로 먼저 구현한다.

#### 이 단계에서 해야 할 일

- 로그인 화면과 회원가입 화면을 만든다.
- 보호 라우트를 적용한다.
- 로그인 후 사용자 정보를 조회해 세션 상태를 유지한다.
- 실패 메시지, 버튼 비활성화, 입력 검증 흐름을 고정한다.
- 백엔드 구현 전까지는 로컬 통신 또는 임시 응답 기준으로 화면 흐름을 먼저 맞춘다.
- 백엔드 인증 API가 준비되면 실제 API 호출로 전환한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| React Router 보호 라우트 | 인증 정보가 없으면 로그인 화면으로 보낸다. 로그인한 사용자가 인증 화면에 다시 들어왔을 때는 팀 목록으로 보내는 흐름도 같이 정한다. |
| TanStack Query | 로그인 이후 `me` 조회처럼 현재 사용자 정보를 확인하는 요청에 사용한다. 인증 여부를 화면 전환 기준으로 활용한다. |
| 로컬 상태 | 이메일, 비밀번호, 이름, 학번 같은 폼 값은 로컬 상태로 관리하고, 제출 시에만 API 요청으로 보낸다. |
| 공통 에러 처리 | 로그인 실패, 회원가입 실패, 필수값 누락 같은 메시지는 화면마다 제각각 쓰지 말고 같은 위치, 같은 톤으로 보여준다. |
| 초기 연동 방식 | STEP 2 초반에는 프론트가 로컬 통신 기준으로 먼저 화면을 완성하고, 백엔드 인증 API가 구현 완료되면 실제 서버 통신으로 바꾼다. |

#### 산출물

- 로그인 화면
- 회원가입 화면
- 보호 라우트
- 인증 상태 유지 흐름

#### 협업 포인트

- BE-A가 반환하는 로그인 응답 필드와 `me` 응답 필드를 먼저 확정해야 한다.
- 토큰 저장 위치와 만료 시 처리 기준은 백엔드와 함께 정한다.
- 서버 초기 셋팅은 STEP 2에서 함께 진행하지만, 프론트는 그 전에 로컬 통신 기준으로 작업을 먼저 진행할 수 있다.

#### 완료 기준

- 비로그인 상태에서 보호 화면 접근 시 로그인으로 이동한다.
- 로그인 성공 후 팀 목록 화면으로 이동한다.
- 새로고침 이후에도 인증 상태가 유지되거나 만료가 정상 처리된다.

### STEP 3

#### 목표

- 팀 관련 기능과 팀 대시보드 1차 화면을 완성한다.

#### 이 단계에서 해야 할 일

- 팀 목록 화면을 만든다.
- 팀 생성 화면을 만든다.
- 참여 코드 입력 흐름을 만든다.
- 팀 대시보드의 기본 카드 구조를 만든다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| TanStack Query | 팀 목록 조회, 팀 생성, 팀 참여 요청을 각각 분리된 요청 단위로 관리한다. 성공 후에는 목록을 다시 불러와 화면을 맞춘다. |
| 공통 레이아웃 | 팀 공간에 들어간 뒤 보이는 사이드바, 상단 헤더, 본문 영역 구조를 여기서 고정한다. 이후 하위 화면은 이 구조를 그대로 사용한다. |
| 카드형 UI | 대시보드는 여러 카드 조합으로 만든다. 공지, 일정, 할 일, 회의록을 카드 단위로 나눠 두면 이후 실제 API 연결 시 교체가 쉽다. |

#### 산출물

- 팀 목록 화면
- 팀 생성 화면
- 팀 참여 UI
- 팀 대시보드 1차

#### 협업 포인트

- BE-A의 팀 목록, 팀 생성, 팀 참여 API 구조와 필드 이름을 먼저 맞춘다.
- BE-B가 만드는 대시보드 보조 조회 결과가 어떤 카드에 들어갈지 미리 정한다.

#### 완료 기준

- 로그인 후 팀 목록이 표시된다.
- 팀 생성 후 생성한 팀이 목록에 보인다.
- 참여 코드 입력 후 팀에 들어갈 수 있다.
- 팀 대시보드 기본 카드 배치가 완성되어 있다.

### STEP 4

#### 목표

- 공지와 일정 도메인을 화면 기준으로 완성한다.
- 첫 번째 AI 기능 연결을 시작한다.

#### 이 단계에서 해야 할 일

- 공지 목록, 공지 작성, 공지 수정 화면을 만든다.
- 일정 목록, 일정 작성, 일정 수정 화면을 만든다.
- 생성, 수정, 삭제 이후 재조회 흐름을 공통 패턴으로 맞춘다.
- 공지 작성 화면에 톤 변환 기능을 연결한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| TanStack Query | 목록 조회 쿼리와 생성, 수정, 삭제 요청을 분리한다. 성공 후 어떤 화면으로 이동할지와 어떤 쿼리를 다시 불러올지 명확히 둔다. |
| 공통 폼 구조 | 제목, 본문, 날짜, 장소 같은 필드는 도메인이 달라도 입력 패턴이 비슷하다. 라벨, 도움말, 오류 메시지 위치를 통일한다. |
| AI 화면 연결 | 공지 작성 화면에서 사용자가 초안을 작성한 뒤 톤 변환을 실행하는 흐름으로 둔다. AI 버튼은 주기능 버튼을 방해하지 않는 위치에 둔다. |

#### 산출물

- 공지 CRUD 화면
- 일정 CRUD 화면
- 톤 변환 UI 연결

#### 협업 포인트

- BE-B의 공지/일정 응답 형식이 UI 리스트 카드와 맞아야 한다.
- AI가 반환하는 톤 변환 결과를 기존 본문에 덮어쓸지, 별도 미리보기로 둘지 기준을 AI 담당과 맞춘다.

#### 완료 기준

- 공지와 일정이 생성, 수정, 삭제까지 전체 흐름으로 동작한다.
- 공지 작성 화면에서 톤 변환 결과를 확인할 수 있다.

### STEP 5

#### 목표

- 할 일과 회의록 도메인을 완성한다.
- 나머지 AI 기능을 실제 화면에 연결한다.

#### 이 단계에서 해야 할 일

- 할 일 목록, 작성, 수정, 상태 변경 흐름을 만든다.
- 회의록 목록, 작성, 수정 화면을 만든다.
- 회의록 작성 화면에 액션 아이템 추출 기능을 붙인다.
- 팀 대시보드에 진행 상황 요약과 팀 문서 Q&A를 붙인다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 상태 표시 UI | 할 일 상태는 색상만으로 구분하지 않고 텍스트를 같이 표시한다. 상태 변경 이후에는 즉시 반영보다 재조회 기준으로 맞춘다. |
| 긴 폼 처리 | 회의록은 입력량이 많으므로 저장 전 임시 입력 상태가 흔들리지 않게 관리한다. 저장 버튼과 AI 기능 버튼 역할을 구분한다. |
| 대시보드 AI 배치 | 진행 상황 요약은 읽기형 카드, 팀 문서 Q&A는 질문과 답변 영역으로 구분한다. 둘 다 대시보드 안의 독립 블록으로 둔다. |

#### 산출물

- 할 일 CRUD 화면
- 회의록 CRUD 화면
- 액션 아이템 추출 UI
- 진행 상황 요약 UI
- 팀 문서 Q&A UI

#### 협업 포인트

- BE-A의 할 일 상태 변경 권한 규칙을 화면에서 막아야 하는지 확인한다.
- BE-B가 Q&A용 문서 조회 데이터를 어떤 형식으로 주는지 AI와 맞춘다.

#### 완료 기준

- 할 일 상태 변경이 실제 목록에 반영된다.
- 회의록 작성 후 액션 아이템 추출 결과를 볼 수 있다.
- 대시보드에서 진행 상황 요약과 팀 문서 Q&A가 동작한다.

### STEP 6

#### 목표

- 배포 환경 기준으로 화면을 정리한다.
- 데모 시나리오를 끊기지 않게 만든다.

#### 이 단계에서 해야 할 일

- 환경별 API 주소를 점검한다.
- 인증 만료, 빈 목록, 오류 응답, AI 실패 응답을 다시 확인한다.
- 데모에서 사용할 주요 화면 순서를 정리한다.
- 실제 배포본 기준으로 브라우저 동작을 점검한다.
- AI 요청이 같은 인스턴스 안의 AI 서버 프로세스를 거쳐도 사용자 흐름이 끊기지 않는지 확인한다.
- NGINX, 로드밸런싱, HTTPS 구성은 프론트 범위에 포함하지 않는다는 점을 확인한다.

#### 어떤 기술을 어떻게 쓰는가

| 기술 | 구현 방식 |
| --- | --- |
| 프로덕션 빌드 | 로컬에서는 되던 경로 이동과 API 주소가 배포 환경에서도 같은지 확인한다. AI 요청의 대기 시간이 조금 늘어도 UX가 흔들리지 않게 점검한다. |
| 오류 점검 | 목록이 비어 있을 때, 요청이 실패할 때, AI 응답이 실패할 때 화면이 깨지지 않도록 최종 확인한다. |
| 데모 흐름 정리 | 로그인, 팀 진입, 공지/일정/할 일/회의록, AI 기능 시연 순서를 하나의 시나리오로 묶는다. |

#### 완료 기준

- 실제 배포본에서 전체 화면 흐름이 끊기지 않는다.
- 데모에 사용하는 주요 기능이 모두 한 번씩 정상 동작한다.
